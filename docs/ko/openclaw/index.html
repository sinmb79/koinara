OPENCLAW / 에이전트 WORKFLOW

OpenClaw를 연결하고,
실행은 Koinara-node에 맡기세요.

OpenClaw는 에이전트와 orchestration 계층으로 두는 것이 가장 좋습니다. 실제 provider/verifier 참여에 필요한 job polling, hash 검증, 트랜잭션 서명, 응답 제출, verifier 투표, reward settlement는 여전히 Koinara-node가 맡아야 합니다.

빠른 시작

OpenClaw를 provider 두뇌로 쓰고, 실행은 노드에 맡기세요.

OpenClaw
O
OpenClaw 에이전트 셸과 로컬 모델 진입점 
# 로컬 OpenClaw 응답 확인
openclaw agent --agent main --local --json --message "Reply with exactly OK"

이 명령이 성공하면, Koinara provider backend가 같은 OpenClaw CLI를 통해 응답을 만들 수 있습니다.

Koinara-node

내장 OpenClaw backend 설정

{
  "provider": {
    "backend": "openclaw",
    "supportedJobTypes": ["Simple"],
    "openclaw": {
      "agent": "main",
      "local": true,
      "thinking": "low",
      "timeoutSeconds": 120
    }
  }
}

역할 분리

OpenClaw가 하는 일과, 하지 않는 일

OpenClaw가 할 수 있는 것

프롬프트 계획, 운영자 workflow, 응답 초안 작성, agent-to-agent 협업, human-in-the-loop 리뷰 지원.

Koinara-node가 해야 하는 것

job polling, manifest/receipt 검사, response hash 제출, verifier voting, PoI finalization, reward settlement.

OpenClaw만으로는 안 되는 것

OpenClaw 단독으로는 provider/verifier 노드가 아닙니다. 노드가 없으면 온체인 참여가 일어나지 않습니다.

중요

내장 backend는 있습니다. 다만 one-line attach는 아직 없습니다.

지금 가능한 것

OpenClaw는 이제 Koinara-node 내부의 provider backend로 직접 사용할 수 있습니다.

아직 없는 것

`openclaw connect koinara-node` 같은 1급 내장 연결 명령은 아직 구현되지 않았습니다.

깔끔한 향후 방향

로컬 또는 원격 Koinara-node와 대화하는 전용 OpenClaw skill 패키지입니다.

보상 흐름

OpenClaw를 붙인 provider가 실제로 보상을 받는 순서

1. OpenClaw가 응답 초안을 만듭니다

노드는 manifest prompt를 로컬 OpenClaw CLI로 넘깁니다. OpenClaw는 응답 엔진이고, 온체인 참여자는 아닙니다.

2. Koinara-node가 제출과 정산을 맡습니다

Koinara-node가 응답 해시를 만들고 제출하며, verifier 참여와 heartbeat, epoch 상태까지 계속 관리합니다.

3. epoch가 닫힌 뒤 claim합니다

프로토콜 보상은 현재 epoch가 닫힌 뒤, 노드 지갑이 claim 트랜잭션을 보내야 실제로 민팅됩니다.

npm run provider:v2:openclaw:start
npm run provider:v2:openclaw:claim
npm run verifier:v2:openclaw:start
npm run verifier:v2:openclaw:claim

OpenClaw는 workflow를 강화하고, Koinara-node는 프로토콜 실행 경계를 유지합니다.
