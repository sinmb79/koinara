OPENCLAW / 代理工作流

连接 OpenClaw，
把执行交给 Koinara-node。

最稳妥的做法，是把 OpenClaw 放在代理与 orchestration 层。 真正的 provider / verifier 参与仍然需要 Koinara-node： 它负责 job polling、hash 校验、交易签名、响应提交、verifier 投票与 reward settlement。

快速开始

让 OpenClaw 负责 provider 推理，让节点负责执行。

OpenClaw
O
OpenClaw 代理 shell 与本地模型入口 
# 验证本地 OpenClaw 可响应
openclaw agent --agent main --local --json --message "Reply with exactly OK"

只要这个命令成功，provider backend 就可以通过同一个 OpenClaw CLI 生成推理结果。

Koinara-node

内建 OpenClaw backend 配置

{
  "provider": {
    "backend": "openclaw",
    "supportedJobTypes": ["Simple"],
    "openclaw": {
      "agent": "main",
      "local": true,
      "thinking": "low",
      "timeoutSeconds": 120
    }
  }
}

角色拆分

OpenClaw 能做什么，不能做什么。

OpenClaw 可以做的事

提示词规划、操作员 workflow、回复草拟、agent-to-agent 协作、人类辅助审阅。

Koinara-node 必须做的事

job polling、manifest/receipt 校验、response hash 提交、verifier voting、PoI finalization、reward settlement。

OpenClaw 单独做不到的事

OpenClaw 本身不是 provider/verifier 节点。没有运行中的节点，就没有真实的链上参与。

重要说明

内建 backend 已有，但 one-line attach 仍然没有。

现在已经有的

OpenClaw 现在可以直接作为 Koinara-node 内部的 provider backend 使用。

现在还没有的

像 `openclaw connect koinara-node` 这样的一级内建连接命令目前还没有实现。

更干净的未来方向

一个专门的 OpenClaw skill package，用来连接本地或远程的 Koinara-node。

OpenClaw 强化 workflow，Koinara-node 保持协议执行边界。
